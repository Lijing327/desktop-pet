# 桌面宠物客户端项目 — Claude Code 执行清单

## 一、项目目标

开发一款面向打工人群体的 **Windows 桌面宠物客户端**，产品形态为本地常驻桌面软件，而不是 Web 页面。

第一阶段目标不是做完整商业化系统，而是先交付一个 **可运行、可演示、可继续扩展** 的 MVP 版本，验证以下核心能力：

1. 宠物可在桌面悬浮显示
2. 支持透明窗口、始终置顶、拖拽移动
3. 支持基础状态切换与简单动画
4. 支持点击互动与随机文案气泡
5. 支持本地提醒能力（如下班提醒、喝水提醒）
6. 支持托盘驻留与基础设置
7. 支持本地数据持久化
8. 可打包为 Windows 安装包或 exe 可执行程序

---

# 二、技术路线约束

## 2.1 技术栈固定

请按以下技术方案实现：

* **客户端框架：Electron**
* **前端界面：Vue 3 + TypeScript**
* **构建工具：Vite**
* **本地存储：electron-store**
* **打包工具：electron-builder**
* **状态管理：Pinia（如有必要）**
* **UI 风格：轻量即可，不要引入过重后台 UI 框架**
* **动画方案：优先使用图片序列帧 / CSS 动画 / 简单状态切换**
* **目标平台：Windows 优先**

## 2.2 架构原则

必须按 **桌面客户端项目** 来设计，不按普通 Web 项目设计。

要求明确拆分以下层次：

1. **Electron 主进程**

   * 窗口管理
   * 托盘
   * 系统通知
   * 开机自启预留
   * 本地能力暴露

2. **Preload 桥接层**

   * 安全暴露有限 API 给渲染进程
   * 禁止渲染进程直接乱用 Node 能力

3. **Vue 渲染层**

   * 宠物视图
   * 对话气泡
   * 设置面板
   * 状态展示
   * 用户交互

4. **本地业务逻辑层**

   * 状态机
   * 提醒调度
   * 文案触发
   * 配置管理
   * 资源切换

# 三、第一阶段交付范围（MVP）

## 3.1 必做功能

### A. 桌宠窗口

实现一个桌面宠物主窗口，满足：

* 无边框
* 透明背景
* 始终置顶
* 不显示任务栏图标（如合适）
* 可拖拽移动
* 默认吸附在桌面右下或左下区域
* 窗口大小固定，后续可扩展

### B. 宠物展示

实现一个基础宠物角色显示组件，支持：

* 默认待机状态
* 行走/轻微动效状态
* 睡觉状态
* 开心状态
* 至少支持 3~4 个可切换状态
* 动画资源结构可扩展，后续能增加更多动作

### C. 点击互动

支持以下互动能力：

* 单击宠物：弹出随机文案气泡
* 双击宠物：触发特殊反馈或切换状态
* 拖拽宠物：可移动位置
* 鼠标悬停：可有轻反馈（非必须）

### D. 气泡文案系统

实现本地随机文案库：

* 支持按分类配置文案
* 至少包含：

  * 打工吐槽类
  * 鼓励陪伴类
  * 提醒类
* 文案以本地 JSON 或配置文件管理
* 触发时显示气泡，并自动消失

### E. 提醒系统

实现本地提醒能力，至少支持：

* 喝水提醒
* 下班提醒
* 午休提醒（可选）
* 可开关
* 可配置提醒时间或间隔
* 提醒时可同时触发：

  * 系统通知
  * 宠物状态变化
  * 文案气泡提示

### F. 托盘功能

实现系统托盘菜单，至少包含：

* 显示/隐藏宠物
* 打开设置
* 暂停提醒
* 退出程序

### G. 设置面板

实现一个轻量设置窗口，至少包含：

* 提醒开关
* 喝水提醒间隔设置
* 下班时间设置
* 是否开机启动（先做预留开关，可后补真实实现）
* 是否启用系统通知
* 文案开关或互动频率配置

### H. 本地持久化

所有用户设置需要本地保存，重启后仍能恢复，包括：

* 宠物位置
* 提醒配置
* 是否隐藏
* 当前状态偏好
* 其他基础设置

### I. 打包交付

支持打包为 Windows 可运行程序，要求：

* 可本地开发运行
* 可生成可分发安装包或 exe
* 给出打包命令和运行说明

# 六、模块设计要求

## 6.1 窗口管理模块

需要支持至少两个窗口：

### 1）宠物主窗口

* 透明
* 无边框
* 始终置顶
* 可拖拽
* 小尺寸悬浮

### 2）设置窗口

* 普通配置窗口
* 可以从托盘打开
* 用于修改提醒和基础偏好

建议对窗口管理做统一封装，不要把窗口创建逻辑散落在入口文件里。

---

## 6.2 宠物状态机模块

请实现基础状态机，不要直接用 if/else 到处切。

至少定义以下状态：

* `idle`：待机
* `walk`：轻微活动
* `sleep`：休眠/发呆
* `happy`：互动反馈
* `remind`：提醒状态

要求：

* 状态切换逻辑可维护
* 支持定时自动切换
* 支持点击触发切换
* 后续可扩展更多状态

---

## 6.3 文案气泡模块

要求支持：

* 文案分类
* 随机抽取
* 显示时长控制
* 避免短时间重复同一条
* 气泡位置跟随宠物附近显示

建议分类：

* `work`
* `comfort`
* `reminder`
* `fun`

示例文案风格可参考：

* 今日 KPI：平安下班
* 你努力工作，我负责陪你
* 该喝水了，再卷也得补水
* 别急，表格不会自己变少

---

## 6.4 提醒调度模块

建议单独封装提醒调度器，支持：

* 周期提醒
* 固定时间提醒
* 开关控制
* 与宠物状态和气泡联动

最少实现：

* 喝水提醒：按分钟间隔
* 下班提醒：按固定时间
* 午休提醒：固定时间，可选

---

## 6.5 本地存储模块

通过 `electron-store` 实现统一存储服务。

要求封装：

* 读取默认值
* 保存配置
* 配置版本兼容
* 首次启动初始化默认配置

建议存储结构至少包含：

```ts
interface AppSettings {
  petPosition: { x: number; y: number };
  reminders: {
    waterEnabled: boolean;
    waterIntervalMinutes: number;
    offWorkEnabled: boolean;
    offWorkTime: string;
    lunchBreakEnabled: boolean;
    lunchBreakTime: string;
  };
  interaction: {
    bubbleEnabled: boolean;
    bubbleFrequency: 'low' | 'medium' | 'high';
  };
  app: {
    launchAtStartup: boolean;
    enableNotification: boolean;
    hideToTray: boolean;
  };
}
```

# 十二、明确限制与注意事项

1. **不要引入复杂后端**

   * 第一版纯本地即可

2. **不要过度追求复杂动画**

   * 先有状态感，再谈精致美术

3. **不要把项目做成网页壳**

   * 重点是桌面客户端体验

4. **不要把设置页做成后台管理系统**

   * 保持轻量

5. **不要一次上太多功能**

   * 第一版只做 MVP

6. **不要为了“看起来高级”引入大量不必要依赖**

   * 优先保证稳定可跑
